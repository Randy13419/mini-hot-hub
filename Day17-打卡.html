Step 2：重试按钮

  • ✅ 失败卡片的「重新加载」按钮 — 点击仅重试该平台，不清除其他平台数据
  • ✅ 后端新增 POST /api/v1/hot-search/retry/:platformId 接口，清除缓存后重新抓取
  • ✅ 重试中按钮显示「重试中…」并禁用，防止重复点击
// 前端调用示例
const handleRetry = async (platformId: string) => {
  const updated = await retryPlatform(platformId);
  // 仅替换失败平台的数据，其他保持不变
  setData(prev => ({
    ...prev,
    platforms: prev.platforms.map(p =>
      p.id === platformId ? updated : p
    )
  }));
};

💬 心得：真实热榜项目和纯展示页有何不同？

纯展示页用的是 Mock 数据，永远不会出错，开发时只关注「页面好不好看」。

真实热榜项目面对的是不可控的上游接口——可能超时、可能改版、可能触发反爬。这次实操中就遇到了 TopHub 返回验证码页面的情况，不得不把知乎适配器切换到官方 API。这让我理解了适配器模式的价值：数据源可以随时替换，只要接口契约不变，上层代码不受影响。

容错是必须的。单个平台挂了不能让整个页面白屏，后端用 Promise.allSettled() 隔离失败，前端按平台独立渲染，加上重试按钮让用户可以主动恢复——这些都是纯展示页不需要考虑的。

合规文案也不是可选项，数据来自第三方平台，上线前必须有清晰的免责声明。
