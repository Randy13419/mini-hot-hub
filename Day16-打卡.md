## 🐛 Bug 修复记录

### Bug 1：B站热度排名错误（Day 16 之前发现）

**现象**：B站卡片排名第4的热度（614万播放）比第1名（253万播放）还高，排名和热度完全对不上。

**原因**：`bilibili.ts` 使用了 `ranking/v2` 接口（全站视频排行榜，按综合得分排序），而不是热搜榜接口。排行榜的排序依据是综合得分，但展示的"热度"是播放量（`stat.view`），两者排序维度不一致。

**修复**：
- 接口从 `ranking/v2` → `search/square?limit=10`（真正的B站热搜榜）
- 热度字段从 `stat.view`（播放量）→ `heat_score`（热搜热度值）
- 标题字段从 `item.title` → `item.show_name || item.keyword`（热搜关键词）
- 链接改为搜索结果页 `https://search.bilibili.com/all?keyword=...`

**修复后验证**：rank 1→10 热度严格递减（369万 → 130万）。

**关键收获**：不同平台"排行榜"和"热搜榜"是完全不同的数据源，API 选错会导致排名逻辑混乱。这正是需要后端中转、适配器隔离的原因之一。

### Bug 2：lastUpdated 缓存期未冻结

**现象**：前端"更新于xx分钟前"永远显示"刚刚"，即使数据实际来自缓存。

**原因**：`controller/hotSearch.ts` 在每次请求时都用 `new Date().toISOString()` 生成 `lastUpdated`，无论数据是否命中缓存。这导致缓存期内时间戳也在不断刷新。

**修复**：
- 引入模块级变量 `lastFetchTime` 记录最后一次实际抓取的时间
- 在并发请求中用 `anyFresh` 标志跟踪是否有任何平台真正重新抓取了数据
- 只有 `anyFresh = true` 时才更新 `lastFetchTime`
- 缓存全命中时，`lastUpdated` 保持上次抓取的时间不变

**修复后验证**：连续3次请求（间隔2秒），`lastUpdated` 完全一致，冻结成功。

**关键收获**：缓存不仅仅是"不重新抓取数据"，还包括"保持与数据对应的时间元信息"。前端显示的"更新于3分钟前"必须与实际数据获取时间一致，否则用户会误以为数据一直在刷新。

## 🧠 深度思考：测试的价值与"防御性编程"

### 为什么真实项目一定要做错误态？

今天的测试过程让我理解了一个关键点：**正常路径只是冰山一角，异常路径才是系统的真正考验。**

在开发阶段，我们默认一切正常——API 能通、数据有返回、网络没问题。但真实世界的用户会遇到各种边缘情况：网络断开、上游接口变更、返回空数据、请求超时。

我们项目中做对的设计决策：

| 设计 | 作用 | 对应 AGENTS.md 约束 |
|------|------|-------------------|
| `Promise.allSettled()` | 单平台失败不拖垮全站 | "单平台失败不影响其他" |
| `withTimeout(5s)` | 防止上游卡死导致请求堆积 | "超时阈值5000ms" |
| 三态 UI (loading/success/error) | 用户始终有视觉反馈 | "每张卡片必须支持三种状态" |
| `MemoryCache TTL=300s` | 保护上游不被频繁请求 | "缓存TTL精确300秒" |
| 前端 catch 异常 | 后端挂掉不白屏 | "后端完全挂掉时前端不白屏" |

这些决策不是"做完功能后再补的"，而是在 AGENTS.md 阶段就预先定义好的约束。**AGENTS.md 本质上就是一份"防御性编程检查清单"**——它在代码生成之前就告诉 AI：这个系统会遇到哪些异常，你该怎么处理。

### B站 Bug 的启示：数据源的准确性验证

B站排名错误的 Bug 给了我一个重要教训：**数据聚合项目的核心风险不在前端展示，而在数据源选择。**

`ranking/v2` 和 `search/square` 是两个完全不同的接口：
- `ranking/v2`：视频排行榜，按综合评分排序（考虑播放、弹幕、投币、收藏等）
- `search/square`：热搜榜，按 `heat_score` 排序（反映实时搜索热度）

用户看到"B站热搜"，预期的是热搜榜，而不是视频排行榜。**接口选错 = 语义错误**，即使代码完全正确，呈现的信息也是错的。

这让我想到：数据聚合项目的测试，不仅要验证"有没有数据"，还要验证"数据对不对"——排名是否符合用户认知、热度值是否合理、数据是否来自正确的数据源。

## 📝 今日总结

Day 16 的核心收获是：**测试不是"检查代码能不能跑"，而是"验证系统在各种场景下是否做了正确的事"。**

通过系统化地跑完测试清单，我发现了3个之前没注意到的 Bug（B站数据源错误、缓存时间跳动、页脚文案缺失）。其中B站的Bug尤其有代表性——功能上"能用"，但数据上是错的，这种Bug最容易被忽视，也最影响用户信任。

> **一句话总结 Day 16**：好的测试清单不只是打勾，而是逼你站在用户视角重新审视每一个细节。
