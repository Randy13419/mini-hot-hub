# 【今日热搜】产品需求文档 (PRD)

> **版本**: V1.0 MVP
> **日期**: 2026-06-06
> **状态**: 待开发

---

## 1. 产品概述与目标 (Product Overview)

### 1.1 产品定位

一个极简、纯净、高效的全网热搜聚合看板。

### 1.2 核心痛点

解决现代人在各个 APP 之间切换带来的**信息焦虑**，打破无穷无尽的推荐算法绑架，将信息获取的主动权交还给用户。

### 1.3 视觉与体验基调 (Vibe)

整体采用类似《文明 7》的高级感 UI 风格：

- 深色沉稳的导航条
- 动态呼吸指示灯
- 卡片式的网格布局
- 排版要求错落有致，简单且高级，**拒绝花哨**

## 2. 核心功能列表 (MVP 优先级 — 本期必做)

> 本 MVP（最小可行性产品）版本必须严格实现以下 5 个核心功能：

### F1. 多平台卡片网格展示 (UI/UX)

| 要素       | 说明                                                         |
| ---------- | ------------------------------------------------------------ |
| 聚合平台   | B站、知乎、抖音                                              |
| 布局方式   | 以平台为维度，每个平台独占一张**竖向卡片**，三张卡片并排网格展示 |
| 交互方式   | 卡片内为可滚动的热搜列表                                     |

### F2. 真实数据解析与智能格式化 (Data Processing)

| 要素         | 说明                                                                                     |
| ------------ | ---------------------------------------------------------------------------------------- |
| 数据源       | 抓取并解析三大平台的**真实**热榜标题与热度值                                              |
| 数字格式化   | 热度超万时使用 **"万"** 为单位（保留一位小数）；不超万时精确到个位                          |
| 动态色彩映射 | 针对不同量级的热度值和数字单位，使用**不同颜色的渐变 UI** 进行标注                          |

**数字格式化示例：**

| 原始热度值  | 格式化结果 |
| ----------- | ---------- |
| `3,847`     | `3847`     |
| `15,230`    | `1.5万`    |
| `102,400`   | `10.2万`   |
| `1,536,000` | `153.6万`  |

### F3. 内存缓存机制 (Performance & Security)

| 要素             | 说明                                                                        |
| ---------------- | --------------------------------------------------------------------------- |
| 缓存策略         | 后端必须实现**内存缓存**                                                    |
| 缓存有效期 (TTL) | 精确 **300 秒**（5 分钟）                                                   |
| 目的             | 避免前端高频访问导致后端频繁请求第三方平台，从而触发防爬虫封禁                |

### F4. 状态与时间更新提示 (Status UI)

| 要素         | 说明                                                   |
| ------------ | ------------------------------------------------------ |
| 更新时间展示 | 页面顶部或显著位置需展示数据的**"最后更新时间"**        |
| 同步机制     | 该时间应跟随缓存的刷新周期同步变动                      |

### F5. 精准跳转与单平台容错 (Interaction & Fallback)

| 要素     | 说明                                                                                      |
| -------- | ----------------------------------------------------------------------------------------- |
| 精准跳转 | 点击任意一条热搜标题，**必须在新标签页 (`_blank`)** 中打开对应的原文链接                    |
| 容错机制 | 若某个平台接口挂掉或超时，该卡片应显示友好的错误提示（如"暂无数据"/"加载失败"），**绝对不能影响其他平台的正常渲染** |

## 3. 后续迭代计划 (Future Scope — 预留扩展窗口，本期不写逻辑)

> 为保证项目顺利落地，明确界定开发边界。在当前 V1.0 版本中，AI 需在**底层架构和 UI 布局**上预留出以下功能的扩展位置，但**绝对不要编写其实际的业务逻辑**：

### 3.1 搜索功能预留（本期不接搜索 API）

- **架构预留**：在顶部导航栏 (Navbar) 的 UI 布局中，为未来的"全局搜索框"预留足够的视觉空间
- **前端组件预留**：预留 `searchQuery` 状态钩子

### 3.2 用户系统预留（本期不接数据库鉴权）

- **架构预留**：在定义数据请求时，考虑到未来可能加入 `userId` 的情况，保持接口调用的独立性
- **UI 占位**：在导航栏右侧预留出"用户头像/登录"的 UI 占位区
- **API 预留**：后端路由统一使用 `/api/v1/` 前缀，为未来鉴权中间件预留挂载位

### 3.3 分类切换预留（本期不做前端 Tab 筛选）

- **架构预留**：在后端抓取和定义热搜数据接口 (JSON 结构) 时，要求尽可能保留或提取原平台的 `category`（分类）字段
- **前端暂缓**：本期前端暂时不渲染筛选 Tab，但底层数据模型必须已具备被分类的能力

### 3.4 新平台接入预留（本期不接其他平台）

- **Adapter 抽象**：后端数据抓取层应抽象出统一的「平台适配器」接口（统一输入输出格式），B站/知乎/抖音各自实现一份适配器。新增平台 = 新增一个适配器文件 + 注册到平台注册表，核心聚合逻辑零改动
- **前端动态渲染**：卡片组件应以平台注册表为驱动动态渲染，而非硬编码三张卡片

### 3.5 历史数据追踪预留（本期不做持久化存储）

- **架构预留**：本期后端暂使用内存缓存（Memory Cache），但缓存层已抽象出 `CacheProvider` 接口。未来若需记录"历史热度曲线"，只需将实现类无缝替换为 Redis 或持久化数据库，核心业务逻辑层零修改

### 3.6 多端与小程序适配预留（本期只做 Web 端）

- **架构预留**：后端 API 采用标准 RESTful JSON 规范输出，严禁与前端视图层强绑定。未来如果项目需要打包成微信小程序或原生 App，可以直接复用这套 API，实现"一处服务，多端调用"

## 4. 附加要求 (合规与兜底)

### 4.1 异常态兜底

必须考虑**"接口通了但无数据"**的空状态 (Empty State) UI，展示友好的占位提示。

### 4.2 接口超时阈值

后端请求单个第三方平台 API 时，应设置明确的超时时间（建议 **5 秒**），避免某个慢平台拖垮整页加载。超时后该平台按容错机制降级处理。

### 4.3 免责声明

页面底部需包含简单的 Footer，声明：

> 数据均来自第三方平台，仅供学习交流使用

### 5.2 数据流说明

```
前端请求 GET /api/v1/hot-search
        │
        ▼
  Controller 收到请求
        │
        ▼
  遍历 PlatformRegistry 中已注册的所有平台
        │
        ├──► 检查缓存 cache:{platform}:hot-search
        │      │
        │      ├─ 命中 → 直接返回缓存数据
        │      │
        │      └─ 未命中 → 调用 Adapter.fetch() (超时 5s)
        │                     │
        │                     ▼
        │               原始数据经 Pipeline 处理
        │               (Formatter → ColorMapper → CategoryExtractor)
        │                     │
        │                     ▼
        │               写入缓存 (TTL=300s)
        │                     │
        │                     ▼
        │               返回处理后数据
        │
        ▼
  Promise.allSettled() 汇总所有平台结果
        │
        │   成功的平台 → 正常数据
        │   失败/超时的平台 → 错误标记 + 降级提示
        │
        ▼
  返回统一 JSON 响应 → 前端按注册表顺序渲染卡片
```

### 5.3 统一数据结构 (JSON 契约)

```jsonc
// GET /api/v1/hot-search 响应结构

  "lastUpdated": "2026-06-06T14:30:00.000Z",  // 最后更新时间，跟随缓存刷新

      "items": [
        {
          "rank": 1,

          "hotValue": "1.5万",   // 已格式化

          "url": "https://...",  // 跳转链接

    }

  ]
}
```

### 5.4 扩展点速查表

| 扩展点 | 当前实现 | 未来可替换为 | 改动范围 |
|--------|---------|-------------|---------|
| 新增平台 | — | 新增 Adapter + 一行注册 | 仅 Adapter 层 |
| 缓存方案 | `MemoryCacheProvider` | `RedisCacheProvider` 等 | 仅 Cache 层实现类 |
| 数据处理 | Formatter + ColorMapper | 插入排序/过滤/聚合等中间件 | 仅 Pipeline 链配置 |
| API 版本 | `/api/v1/` | `/api/v2/` 并行存在 | 仅路由层 |
| 鉴权中间件 | 无 | 挂载到 `/api/v1/` 路由组 | 仅中间件注册 |
| 前端卡片 | 3 张卡片动态渲染 | N 张卡片（平台注册表驱动） | 无改动 |
